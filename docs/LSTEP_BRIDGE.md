# L ステップ共存 / Bridge 設計

L ステップを使い続けたい顧客に L-port の AI 機能を載せるための共存設計と、営業時の出し分け。

> 前提となる LINE の根本仕様: **1 つの LINE 公式アカウント = Webhook は 1 つだけ**。
> ユーザーから届く受信イベントは、たった 1 つの URL にしか飛ばない。
> → 同一 OA で「L ステップ」と「L-port」が両方とも独立して受信を握ることはできない。
> 「自動応答だけ L-port で」が成立するかは **受信 (Webhook) を誰が持つか** で決まる。

---

## 3 つの構成パターン

### パターン A: L-port が Webhook を直接握る (AI 自動応答プランの王道)

- OA の Webhook → L-port。AI 自動応答は L-port が担当。
- 一斉配信は問題なく共存可 (配信は push/broadcast API で Webhook を使わないため)。お客さんは L ステップでも L-port でも配信できる。
- ただし **L ステップの受信トリガー系自動化** (キーワード応答・回答フォーム反応・流入タグ付け・ユーザー操作起点のステップ配信) は動かなくなる (Webhook を取られるため)。
- → 顧客の L ステップ利用が「一斉配信 + 手動運用が中心」なら、これでキレイに回る。
- **コスト: reply トークンでの自動応答は無料・無制限。** A が最安。

### パターン B: L ステップが Webhook を握り、L-port へ転送 (← 実在機能で組める)

- **2025-06 に L ステップが「Webhook 転送機能」を正式リリース済み**（前回「無い認識」としていたが覆った）。
  - 月額 **5,500 円(税込)** オプション / プロプラン以上が前提（顧客負担）。
  - 流れ: LINE 公式 → L ステップ → 指定した外部 URL に JSON を POST。
  - 対象イベント: 友だち追加 / **メッセージ受信** / フォーム回答 / タグ操作 / 商品購入 など。
- **L ステップの自動化を全部残したまま** L-port の AI 応答を共存できる。
- ただし L-port は reply (応答トークン) が使えず **push API で返す** ことになる。
  - ⚠ コスト面: reply は無料・無制限だが、**push は月間メッセージ通数を消費**する。
  - つまり B は「動いても従量コストがかかる」構図。+5,500 円/月 と push 通数が顧客負担。
- 受信側エンドポイント (L-port 実装):
  `POST https://api.line-port.com/api/lstep/webhook/{accountId}`

### Bridge (タグ同期どまり)

- L ステップが受信・配信・シナリオを握ったまま、L-port は AI 判定したセグメントタグ等を API で書き戻すだけ。
- この構成だと L-port は**リアルタイム受信を持たない**ので、AI 自動応答そのものは担えない
  (担えるのはトーク履歴ポーリング + push 返信という遅延・有料の代替策のみ)。
- `lstep-client.ts` の双方向同期 (友だち / タグ / 対応マーク / トーク履歴) がこれ。Webhook の横取り/中継ではない。

---

## 出し分け (営業)

| 顧客の L ステップ利用 | 推奨構成 | 必ず説明すること |
|---|---|---|
| 一斉配信中心 + 手動運用 | **パターン A** | L ステップの受信トリガー自動化は止まる |
| キーワード応答/ステップ配信など自動化もガッツリ + AI も欲しい | **パターン B** | +5,500 円/月、AI 返信は push 従量 |
| 既存運用を一切触らず AI 分析だけ足したい | **Bridge** | リアルタイム AI 応答は不可 (タグ同期のみ) |

ヒアリングの肝: 「L ステップを一斉配信中心で使ってますか？それともキーワード応答・ステップ配信など **Webhook 依存の自動化** までガッツリ使ってますか？」 ← ここで A / B / Bridge が決まる。
