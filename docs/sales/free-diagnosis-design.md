# L-port 無料診断「LINE運用度診断」設計書

> **実装メモ（2026-06-17）**: Web 入口は当初 `apps/web` を想定していたが、実装では
> **line-port.com を配信する l-port-lp（独立 LP）側の静的ページ `diagnosis.html`** に変更した。
> LIFF 引き継ぎは `apps/liff` ではなく **worker クライアント（`apps/worker/src/client/main.ts` の `?page=diagnosis`）** で実装。
> LP には無料診断 CTA を追加し、問い合わせフォームを `POST /api/inquiries`（`source: 'lp_free_consult'`）へ実配線済み。
> 診断フォーム seed = `migrations/081_seed_free_diagnosis.sql`（form_kind='diagnosis'）。本番/staging 両 D1・両 worker へ反映済み。

## 0. 目的・位置づけ

L-port（DfY 型 AI LINE 運用代行）のリードマグネット（見込み客獲得の入口）。
見込み客が自社の LINE 運用を自己診断 → スコア＋ボトルネックを提示 → 「詳細レポートは LINE で」で友だち化 → ナーチャ → 無料ヒアリング(30分)予約、までを一本の動線にする。

競合（Liny「LINEマーケ力診断」、commercepick「運用度チェックシート」等）の定番＝**友だち追加→3〜5問→タイプ/レベル別結果→相談誘導**。本診断はこれを踏襲しつつ、結果を **運用度スコア（型1）＋最大ボトルネック特定（型2）のハイブリッド** にして「で、何をすればいいか」まで提示する。

---

## 1. 動線（Web入口 → LINE誘導）

```
[LP / 広告]
   │  「無料LINE運用度診断（30秒）」CTA
   ▼
[Web診断ページ（公開・匿名OK）]  ← 業種選択 + 6問の選択式
   │  即時に簡易結果を表示
   ▼
[Web簡易結果]  運用度スコア(100点/レベルA-D) ＋ 最大ボトルネック1つ
   │  「業種別の詳細改善レポート＋具体策はLINEで受け取る」
   ▼
[LINE友だち追加CTA]  ← ここでリード化（friend_add）
   │  （回答は querystring / ref で引き継ぎ）
   ▼
[LIFF or トーク内]  回答を引き継いでフォーム送信
   │  POST /api/forms/:id/submit
   ▼
[結果Flex配信]  スコア / ボトルネック / L-portでの解決策 / 「無料ヒアリングを予約」ボタン
   │  + segment_tag付与 + metadata保存 + on_submit_scenario_id でナーチャ登録
   ▼
[ナーチャシナリオ]  結果解説 → 業種別事例 → ヒアリング予約催促（数日のステップ配信）
```

設計上のポイント:
- Web で「スコアの一部＋ボトルネックの見出しだけ」見せ、**詳細（改善策・業種別レポート）は LINE 登録後**に出す＝間口は広く、リード化は確実に。
- 回答の引き継ぎは URL パラメータ（例 `?ans=2,1,3,0,2,1&industry=salon`）で LINE 誘導 → LIFF が読み取り自動送信、ユーザーは LINE 側で再入力不要。

## 2. 設問（確定）

冒頭に業種選択（美容室 / 整体・治療院 / EC・D2C / スクール・教室 / 士業 / 飲食店 / その他）。
以下 6 問、各 4 択。配点は選択肢ごとに **0 / 1 / 2 / 3 点**。

### Q1. 友だち獲得導線
「LINE の友だちを増やす仕組みは、どれくらい整っていますか？」
- 3点: 店頭・Web・広告など複数の導線があり、毎月安定して増えている
- 2点: 店頭QRなど1つの導線で、少しずつ増えている
- 1点: 友だち追加をお願いしているが、ほぼ増えていない
- 0点: 特に何もしていない / 友だちがほとんどいない

### Q2. 初動フォロー（追加直後）
「友だち追加してくれた人に、すぐ使いたくなるメッセージやクーポンを送っていますか？」
- 3点: あいさつ＋特典を自動で送り、初回来店/購入まで設計している
- 2点: あいさつメッセージは自動で送っている
- 1点: たまに手動で送ることがある
- 0点: 何も送っていない

### Q3. リッチメニュー
「トーク画面下のリッチメニューは、お客様が使いやすいよう設計されていますか？」
- 3点: 目的別に整理し、対象によって出し分けている
- 2点: 設置していて、予約や問い合わせに使えている
- 1点: 設置しているが、ほぼ使われていない
- 0点: 設置していない

### Q4. セグメント / パーソナライズ配信
「配信内容を、お客様の属性や興味に合わせて出し分けていますか？」
- 3点: タグ/セグメントで出し分け、一人ひとりに合った内容を送れている
- 2点: 一部のお客様を分けて送ることがある
- 1点: 全員に同じ内容を一斉配信している
- 0点: そもそも配信していない

### Q5. リピート施策
「LINE上でリピートを生む仕組み（回数券・スタンプ・予約・会員証など）はありますか？」
- 3点: 複数の仕組みがあり、リピート/再来が回っている
- 2点: 1つはある（例: クーポンや予約）
- 1点: 用意したいが手をつけられていない
- 0点: 何もない

### Q6. 効果測定
「配信やクーポンの成果（クリック・予約・売上）を計測して改善できていますか？」
- 3点: CV や流入を計測し、数字を見て改善している
- 2点: 開封やクリックは見ている
- 1点: 送りっぱなしで結果は見ていない
- 0点: 計測の仕方がわからない

## 3. 採点ロジック（一意に決まる）

- 生スコア = Q1〜Q6 の合計（0〜18点）
- 100点換算 = `round(生スコア / 18 * 100)`
- レベル判定:
  - **A（80点以上）**: 運用上級者。L-port で「伸ばす/手離れ」フェーズ
  - **B（60〜79点）**: 基礎はできている。あと一歩で成果が変わる
  - **C（40〜59点）**: 仕組みが部分的。取りこぼしが多い
  - **D（39点以下）**: ほぼ未活用。伸びしろが最大
- **ボトルネック軸 = 6軸のうち最低スコアの軸**。
  - 同点の場合は売上インパクト順で決定: **Q6 効果測定 > Q1 友だち獲得 > Q5 リピート > Q2 初動 > Q4 パーソナライズ > Q3 リッチメニュー**
  - （理由: 計測できていないと全施策が改善不能 → 入口の友だちがいないと何も始まらない → 既存客のリピートが利益直結、の順）

## 4. ボトルネック → 結果文言 ＆ L-port 解決策（型2）

各ボトルネックに対し「一言診断」＋「L-port のこの機能で解決」を提示する。

| ボトルネック | 一言診断 | L-port での解決策（機能） |
|---|---|---|
| Q1 友だち獲得 | 入口が細く、母数が増えていない | トラッキングリンク/流入経路計測、トラフィックプール（複数アカ自動振り分け）、QR動線設計 |
| Q2 初動フォロー | 追加直後の熱が冷めて離脱している | friend_add トリガーのステップ配信、あいさつ＋初回クーポンの自動化 |
| Q3 リッチメニュー | 一番見られる場所が活かせていない | タグ別/ユーザー別リッチメニュー自動切替 |
| Q4 パーソナライズ | 全員に同じ配信でブロックを招いている | タグ/セグメント配信、AI接客（個別レコメンド） |
| Q5 リピート | 新規は来るが再来の仕組みがない | クーポン/回数券、Web予約、リマインダー配信 |
| Q6 効果測定 | 改善の根拠がなく勘で運用している | CV計測、分析ダッシュボード（結果が見える管理画面） |

**全レベル共通の締め（型1→本体価値への接続）**:
「これらを“自分で運用する”のは本業を圧迫します。L-port なら専属チーム＋AIが月¥19,800〜で全部代行。まずは無料ヒアリング(30分)で、あなたの店に合う改善プランをお出しします。」→ ヒアリング予約ボタン。

## 5. 結果パターン（スコア帯 × ボトルネック）の出し分け

- 見出し = レベル（A〜D）の総評（4種）
- 本文 = ボトルネック軸の一言診断＋解決策（6種）
- CTA = 全レベル共通のヒアリング予約

→ 4×6 を個別に書き起こす必要はなく、**「レベル総評（4文）」＋「ボトルネック文（6文）」の組み合わせ**で 24 パターンを表現する（メンテしやすい）。

### レベル総評（4文・確定）
- **A**: 「運用レベルはかなり高いです。あとは“手離れ”と更なる最適化が次の一手。」
- **B**: 「基礎はできています。弱点を1つ潰すだけで成果が大きく変わる段階です。」
- **C**: 「仕組みが部分的で、取りこぼしが多い状態。優先順位をつければ伸びます。」
- **D**: 「LINEがほぼ未活用。逆に言えば伸びしろが最大。今が始めどきです。」

## 6. データ設計（既存基盤に載せる）

- フォーム定義: `forms.fields`(JSON) に Q1〜Q6＋業種を格納。各選択肢に配点を持たせる（`options[].score`）。`save_to_metadata=true`。
- 回答保存: `form_submissions.data`(JSON)。`friends.metadata` に業種/スコア/レベル/ボトルネックをマージ。
- セグメント付与: `friend_segment_tags` に「業種」「スコア帯(レベル)」「ボトルネック種別」を付与（`assigned_by='ai'` 相当、`reason` に診断由来と記録）。
- シナリオ: `forms.on_submit_scenario_id` でナーチャ登録（`enrollFriendInScenario`）。
- 結果Flex: submit 後に既存の結果メッセージ配信フローへ。
