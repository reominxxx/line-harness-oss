# L-port チーム共有メモ (2026-05-31 更新)

「全部こっちが運用代行する」というモデルに変えたので、URL も画面も整理し直しました。
これからの仕事の進め方をシンプルにまとめます。

---

## 🌐 URL は 4 つだけ覚えればいい

| URL | 用途 | 触る人 |
|---|---|---|
| **https://line-port.com** | LP (会社案内・営業ページ) | 見込客に渡す |
| **https://app.line-port.com** | 顧客ダッシュボード | 契約済みの顧客に渡す |
| **https://team.line-port.com** | 運用チーム管理画面 | **私たちだけ** ※顧客に絶対渡さない |
| https://api.line-port.com | バックエンド API | 触らない (LINE Webhook 用) |

---

## 👥 役割と画面の対応

### お客様 (店舗オーナー)
**渡す URL**: `https://app.line-port.com`

お客様は **これしか触らない**。中身はシンプル:
- 自分の店の **配信実績** が見える
- **クーポンの使用率** が見える
- **友だち数の推移** が見える
- **AI 接客件数** が見える
- 月次レポートが届く
- チャットで質問できる

→ お客様には **「app.line-port.com にアクセスしてください」とだけ伝える**。

### 私たち (運用代行チーム)
**使う URL**: `https://team.line-port.com`

ここで毎日の運用を回す:
- 配信文を作成 / 予約 / 配信
- クーポン作成・配信
- リッチメニュー設定
- AI 設計書生成 (ヒアリング録音から月 N 本の設計書出力)
- 友だちタグ管理
- 設定 (アカウント追加・トークン管理)

→ **顧客には絶対に team.line-port.com の URL を見せない・教えない**

---

## 🛡 安全装置 (誤って漏れても大丈夫な作り)

万が一お客様に team URL を伝えてしまっても、以下の二重ガードで守られます:

1. **物理分離**: team.line-port.com と app.line-port.com は別ドメイン。お客様のログインアカウント (`customer` role) では team 側にログインできない
2. **API レベル拒否**: お客様の認証情報で team URL から API を叩いても、Worker 側で `403 Forbidden` を返す
3. **画面リダイレクト**: お客様が `app.line-port.com/friends` のような admin URL を打っても、自動で `/client/login` に飛ぶ

## 🚀 新規顧客のオンボーディング (標準フロー)

### Step 1: 営業 (LP からの問合せ)
- 見込客が https://line-port.com から問合せフォーム送信
- 30 分の無料ヒアリング (Zoom)

### Step 2: 契約 + 認証情報のご共有
- 契約後、顧客に以下を聞く:
  1. LINE 公式アカウントのチャネル ID
  2. チャネルアクセストークン (長期)
  3. チャネルシークレット
  4. (任意) 既存タグ・友だち CSV
- → **対面 / 暗号化チャットで受け取る** (メール添付は禁止)

### Step 3: team.line-port.com でセットアップ (我々がやる)
1. `https://team.line-port.com/login` でログイン
2. 左メニュー **「アカウント運用」** → 新規アカウント追加
   - 顧客の LINE channel 情報を入力
3. 左メニュー **「友だち管理」** → 「🔄 LINE と同期」で既存友だちインポート
   - 取れない場合 (プラン制限) は、顧客に「リッチメニュー押して」と伝える
4. **「設計書」** で初回ヒアリング録音をアップロード → 月 N 本の配信プラン自動生成
5. **「一斉配信」** で配信案作成 → スケジュール
6. **「クーポン」** で初月分のクーポン設計
7. **「リッチメニュー」** で店舗用メニュー設定
8. **「AI 配信設定」** で業界プレイブック投入

### Step 4: 顧客アカウントの発行 (お客様用ログイン作成)
1. **team.line-port.com** → 左メニュー **「スタッフ管理」** (※ 顧客もここで作成、role=customer で)
2. 「顧客アカウント追加」
   - 顧客のメールアドレス
   - 役割: **customer**
   - 紐付け LINE アカウント: 上記で追加したアカウント
   - パスワード自動生成 → 顧客にメールで送付

### Step 5: 顧客への引き渡し
顧客に以下を送る:

```
件名: L-port のお客様画面ご案内

○○ 様

LINE 運用のご契約ありがとうございます。
お客様専用ダッシュボードのご案内です。

URL: https://app.line-port.com
メールアドレス: (顧客のメール)
パスワード: (発行したパスワード)

このページでは:
・配信実績
・クーポン使用状況
・友だち数の推移
・AI 接客件数
・月次レポート
が確認できます。

ご質問はチャットからお気軽にどうぞ。
配信や設定の作業はすべて L-port チームが行いますので、
お客様はレポートを見るだけで OK です。
```

## 📅 日々の運用 (チーム作業)

### 毎日 (朝 10:00 まで)
- team.line-port.com → **「未対応」** に新着問合せがあるか確認
- AI が答えられないものに人間で返信

### 週次 (月曜午前)
- team.line-port.com → **「設計書」** で今週分の配信案を承認 → 予約
- **「分析」** で前週の数字確認、必要なら改善 (タグ追加 / セグメント変更 等)

### 月次 (月初 1 営業日)
- 月次レポート生成 (自動)
- 顧客に「先月の振り返り + 今月の方針」を Slack / メールで送付
- Pro プラン顧客は Zoom 30 分の戦略 MTG

## ⚠️ やってはいけないこと

| やめて | 理由 |
|---|---|
| 顧客に team.line-port.com を伝える | 顧客が admin に入れちゃう → 設定壊される / 他社情報漏洩 |
| 顧客 LINE アカウントで配信テスト | 実顧客に「テストです」が届く → クレーム |
| 直接 D1 を触る | データ消すと復旧難しい。管理画面経由で操作する |
| 本番 worker の secrets を変更 (要事前確認) | LINE 接続が切れる |
| GitHub に secrets / API key / トークンを commit | 漏洩リスク、誰かに警告される |

## 🔧 staging 環境 (動作テスト用)

新しい機能を試したい / 配信フローをテストしたい時は **staging** を使う:

- `https://staging.line-port.com` → 検証用 admin / 顧客画面
- `https://staging-team.line-port.com` → (これも staging チーム画面、CNAME 後で追加予定)
- 本番とは **完全に別 DB / 別 LINE アカウント**
- Cron は止まってる (実配信されない)
- 画面上部に黄色の「⚠ STAGING ENVIRONMENT」バナー表示

新機能をリリースする時は:
1. ローカル / staging で動作確認
2. team 内 review
3. 本番デプロイ

## 🔐 セキュリティのルール

1. **2FA 必須** (今後設定): Cloudflare アカウント / Anthropic / LINE Developers
2. **顧客の認証情報は秘密管理**: 1Password / Bitwarden 等に保存、平文置かない
3. **トークン漏洩したら即 rotate**: `npx wrangler secret put XXX --config ./wrangler.toml`
4. **デプロイは慎重に**: 本番 deploy は `./scripts/deploy-prod.sh` (確認プロンプト付き)、staging は `./scripts/deploy-staging.sh`
5. **コードレビュー**: 大きな機能追加は誰かにレビュー依頼してから本番反映

## 🆘 困ったとき

| 症状 | 確認手順 |
|---|---|
| 顧客の管理画面に入れない | team.line-port.com → スタッフ管理 → 該当顧客のパスワード再発行 |
| LINE Webhook が来ない | LINE Developers → 該当 channel → Webhook URL が `https://api.line-port.com/webhook` になってるか |
| 配信が送られない | team の「一斉配信履歴」で status 確認、cron が 5 分以内に動くまで待つ |
| 設計書が「設計中」で止まる | DB で hearings の status='generating' が 5 分続いたら、cron が自動で pending に戻す。それでも進まなければ Anthropic API key を確認 |
| ドメインが見えない | DNS 伝播待ち、5-30 分は待つ。dscacheutil -flushcache でローカルキャッシュもクリア |
